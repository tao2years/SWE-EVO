# 仓库协作初始化约定

本文件用于记录仓库级 `/init` 协作约束，默认作为后续开发、清理、迁移和路线图推进时的共同执行基线。

## 输出语言

- 默认使用中文沟通、记录和收尾。
- 如需英文，仅在用户明确要求时切换。

## 文档落点

- 主线规划优先落在 `docs/roadmap/`。
- 跨多轮执行的任务、迁移、清理和阶段性交付计划落在 `docs/exec-plans/`。
- 纯技术债、待处理问题或暂不执行事项，应回推到路线图、执行计划或独立技术债记录，不在会话里口头悬空。

## 执行与路线图

- 主线任务先重述目标
  - 用户要求“对齐路线图 / 继续主线”时，先说明当前主目标、阶段和下一刀。
- 每一刀都要可追踪
  - 改动要么回推到 `docs/roadmap/`，要么登记到 `docs/exec-plans/` 或技术债追踪。
- 清理不能替代交付
  - 连续两轮主要在做治理减后，下一轮优先回到未完成主线。
- 长任务必须落计划
  - 超过一轮的实现、迁移、清理，写入 `docs/exec-plans/` 并持续更新进度日志。
- 主线冲突先清障，不保旧面
  - current 规划与旧实现直接冲突时，先删或下线阻挡主线的旧页面、旧命名、旧文档，再继续实现；不要为了“看起来兼容”保留双轨。
- 默认不为顺手问题偏航
  - 已经选定本轮主线后，除非该问题直接阻塞当前交付、会让新改动变假配置/假入口，或用户明确要求，否则不要切去处理支线优化、额外治理、零引用清理或“顺手修一个”。
- 清理必须有主线收益句
  - 任何治理/重构/删减动作，动手前都要能用一句话说明“它如何直接帮助当前主线交付”；如果说不出来，就记录为后续项而不是立即执行。
- 顺手项一次只收一刀
  - 实现主线时即使发现多个周边问题，默认只处理其中最直接阻塞的一项；其余登记后立即回到主线，不串行深挖。
- 完成判定先看主线，再看周边
  - 用户问“完成了么”时，先回答主线目标是否完成；周边清理、额外校验、可选优化必须单标为“已做 / 未做”，不能混淆“还差一点边角所以整体未完成”。
- 验证以证明交付为上限
  - 检验应先覆盖当前改动的真实风险；在已经证明主线可交付后，不要因为还能继续跑更重检查，就无限追加验证并拖延收口。
- 开发任务未必须给完成度百分比
  - 非纯问答的开发任务收尾时，必须给“本轮完成：X%”，并说明主线目标是否完成、验证情况、剩余缺口和下一刀；路线图 / 长任务 / 多阶段主线还要额外给“整体目标完成度：Y%”，并说明百分比口径。
