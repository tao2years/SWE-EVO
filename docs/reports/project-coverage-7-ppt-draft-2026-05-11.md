# project-coverage-7 对比汇报 PPT 底稿（2026-05-11）

## 用途

这份文档不是完整复盘报告，而是给 1 到 2 页 PPT 直接取材的底稿。  
目标对象是：

- `pc7-claude-2.1.89`
- `pc7-innercc-init`
- `pc7-innercc-context`
- `pc7-claude-2.1.138`

核心回答 3 个问题：

1. 引入 context 相关特性以后，提升到底在哪里。
2. `2.1.89` 和 `init` 这个“理论持平版本”是否真的持平。
3. `2.1.138` 最新版为什么又多了 `1` 个 case。

## 第 1 页

### 页标题

**project-coverage-7 四版本结论：context 的提升集中在 1 个大 case，2.1.138 的新增收益集中在 1 个小 case**

### 建议版式

- 左侧一张 4 列对比表
- 右侧 3 条结论

### 右侧结论文案

#### 结论 1

`2.1.89` 和 `init` 在 benchmark outcome 上是**逐案完全一致**的：

- `resolved` 一样：`3/7`
- `f2p micro` 一样：`10/22`
- `p2p micro` 一样：`4969/9156`
- 7 个 case 的 `resolved/F2P/P2P` 结果逐项一致

但它们**效率不一致**：

- `2.1.89` 比 `init` 多 `166` turns
- 成本高 `~$14.8`
- 输入 token 多 `~3760k`

结论要点：

- 可以说“效果持平”
- 不能说“过程和效率也持平”

#### 结论 2

`context` 版本的提升不是平均分布在所有 case 上，而是**几乎全部集中在 `pydantic__pydantic_v2.7.1_v2.7.2`**：

- `F2P`: `0/3 -> 3/3`
- `P2P`: `403/4584 -> 4574/4584`
- run-level `p2p micro`: `54.27% -> 99.83%`

其余 case 基本没动：

- `dask`、`modin`、`requests`、`conan`、`scikit` 结果没变
- `iterative` 仍未 resolve

#### 结论 3

`2.1.138` 比 `2.1.89/init` **只多 1 个 resolved case**：

- `resolved: 3 -> 4`
- 新增的是 `iterative__dvc_1.0.0b6_1.0.0`

它不是全面变强，而是：

- 大坑 `pydantic` 仍然没解决
- 只是把一个“小而强契约”的 case 终于补通了

### 这一页建议讲稿

可以直接讲：

> 先看总盘子。  
> 第一，2.1.89 和 init 不是“大体接近”，而是这 7 个 case 的最终 outcome 完全一致，所以就这个子集来说，理论持平在结果上是成立的。  
> 第二，context 版本的提升非常集中，不是全面涨，而是几乎全靠 pydantic 这一案把 F2P 和 P2P 一起拉起来。  
> 第三，2.1.138 确实更好，但不是全面更好，只是比 2.1.89/init 多解了 1 个 case。  
> 所以这页的主线是：结果差异要按 case 看，不能按 run-level 平均理解。

## 第 2 页

### 页标题

**根因拆解：context 为什么提升，2.1.138 为什么多 1 个 case，哪些结论能说，哪些不能过度说**

### 建议版式

- 上半页：3 个 driver cases
- 下半页：一句“能下的结论”和一句“不能过度归因的结论”

### 上半页核心表格

| --- | --- | --- | --- | --- | --- |
| `pydantic__pydantic_v2.7.1_v2.7.2` | `0/3`, `403/4584` | `0/3`, `403/4584` | `3/3`, `4574/4584` | `0/3`, `403/4584` | 决定 `p2p micro`，也决定 `context` 的主要提升 |

### 下半页文案

#### A. 为什么引入 context 以后有提升

实事求是地讲，当前证据支持两层判断：

第一层，**结果层面确实提升了**：

- 只看 `pc7` 这 7 个 case，`context` 把 `pydantic` 这 1 个大坑从：
  - `0/3 F2P, 403/4584 P2P`
  - 拉到
  - `3/3 F2P, 4574/4584 P2P`

第二层，**这次提升不能简单全部归因给 context prompt 本身**：

- `context` 那次评测日志里，`typing-extensions` 成功从 `4.10.0` 升到了 `4.13.2`
- 随后 pytest 从 `collected 1019 items / 57 errors` 变成 `collected 5150 items`
- `NoDefault` 不再在 Python 3.8 环境里 import 失败

所以更准确的说法是：

- `context` 版本这次 run 的最终结果显著更好
- 但这个提升同时包含了**上下文能力**和**依赖环境成功更新**两部分
- 当前证据不足以把 `99.8% p2p` 全部说成 prompt 特性本身的纯收益

#### B. `2.1.89` 和 `init` 理论持平，是否确实一致

结论：

- **结果一致**
- **效率不一致**

结果一致的证据：

- 7 个 case 的 `resolved/F2P/P2P` 逐项完全相同

效率不一致的证据：

- `2.1.89` 成本更高、turns 更多、输入 token 更多

因此汇报时最严谨的说法应该是：

- “2.1.89 与 init 在这个子集上的 benchmark outcome 确实一致”
- “但 2.1.89 没有表现出更好的效率或更稳的收敛过程”

#### C. `2.1.138` 为什么多了 1 个 case

新增的那 1 个 case 是：

- `iterative__dvc_1.0.0b6_1.0.0`

根因不是环境偶然变好，而是修复策略更对题了：

- `2.1.138` 直接把 `no_exec=self.args.no_exec` 作为公开接口契约传到 repo 层
- 这正好命中 benchmark 的 unit test / mock contract
- 而 `2.1.89/init` 都没有把这题真正收口

更直白一点：

- `2.1.138` 这次多出来的不是“大题能力”
- 而是“强契约小题终于补通”

### 这一页建议讲稿

可以直接讲：

> 第二页只讲 driver cases。  
> context 的提升点在哪里？答案是非常集中，几乎全在 pydantic 这一案。  
> 但这里要实事求是：这次提升不是只靠 prompt，安装链路里 `typing-extensions` 成功升级也是关键条件。  
> 再看 2.1.89 和 init，它们最终分数完全一样，所以“理论持平”在结果上成立，但 2.1.89 更贵、更慢。  
> 最后看 2.1.138，它比前面多出来的那个 case 不是 dask，因为 dask 在 2.1.89 已经过了；真正新增的是 iterative，这说明最新版的进步点主要在强契约 case，而不是已经把大坑 pydantic 解决了。

## 汇报时建议明确说的 3 句话

1. **可以明确说：** `2.1.89` 和 `init` 在这个子集上的最终效果确实一致，而且是一致到逐 case outcome 完全相同。  
2. **可以明确说：** `context` 版本这次的提升几乎全部集中在 `pydantic`，提升点不是平均分布在所有 case 上。  
3. **必须保留限定：** `context` 的大幅提升不能全部归因给 prompt/context 特性本身，因为依赖环境成功更新是关键条件之一。

## 汇报时不要直接说的 2 句话

1. **不要直接说：** “context 让模型全面变强了。”  
   更准确的是：这次 run 里，`context + 成功的依赖更新` 一起让 `pydantic` 这案显著改善。

2. **不要直接说：** “2.1.138 全面优于 2.1.89。”  
   更准确的是：`2.1.138` 在这个子集上多解了 `iterative` 这 1 个 case，但 `pydantic` 这个大坑仍然没过。
