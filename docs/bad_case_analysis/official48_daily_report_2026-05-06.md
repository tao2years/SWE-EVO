## 核心结论

- 全量 `48` 案已经看完，当前不是“少数 bad case”，而是足够下阶段性结论的样本量。
- 当前主问题不是代码生成能力不够，而是任务拆分、定位、验证、结束策略这一整条收敛链经常失真。
- 最大共性问题是验证没收口：`validation_gap = 41/48`。
- 第二层问题是任务理解错位：`task_understanding_error = 30/48`。
- 第三层问题是过早锁定第一个看起来像的解释：`hypothesis_lock_in = 27/48`。

## 两套 CLI 的差异

- `innercc` 在大 bundle、多子任务、多层联动题上更容易碰到 benchmark 主轴。
- `innercc` 的代价也很明显：更容易被环境噪声带偏，补丁一旦改宽，P2P 回归会更重。
- `claude-code` 在小题、单点题、强契约题上更稳，尤其是参数语义、接口行为、返回格式这一类任务。
- `claude-code` 的主要问题是大题上容易过早收缩，只修到最显眼的一小块。

## 可直接汇报版

- 截至 `2026-05-06`，`official48` 的 `48/48` 个 case 已全部完成白话化复盘。
- 整体看，当前主要瓶颈不是“模型不会写代码”，而是“大题没拆开、定位过早锁死、验证不够硬”。
- `innercc` 更适合多子任务和大 bundle，但更容易被环境噪声带偏，补丁扩散后回归更重。
- `claude-code` 在单点和强契约题上更稳，但大 bundle 上常常只修到最显眼的一小块。
- 现在最值得沉淀成工程规则的，不是某个 repo 技巧，而是统一的任务拆分和验证门槛。

## 下一步建议

- 前几步就强制判断任务规模：单点题、多子任务题、bundle 题分开处理。
- 用 `FAIL_TO_PASS` 分布先做目标聚类，禁止把超大 bundle 直接当单点 bug 修。
- 结束前强制做两层验证：exact F2P 必过，相邻 P2P 至少做最小 smoke。
- patch 覆盖面和任务规模明显不匹配时，不允许直接结束。
